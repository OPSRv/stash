# stash-separator runtime deps. uv resolves these inside the venv at
# `Settings → Separator → Install` and the resulting interpreter is
# what the in-app job spawns.
#
# Notes on the pins:
# - `demucs==4.0.1`: the API surface main.py uses
#   (`demucs.pretrained` + `demucs.apply` + `demucs.audio`) is stable
#   from 4.0.1 onwards. The newer `demucs.api.Separator` wrapper is
#   only on git head and isn't in PyPI 4.0.1, so we don't depend on
#   it. Pinning explicitly takes resolver ambiguity off the table.
# - `librosa`: drives BPM detection via `librosa.beat.beat_track`
#   on the separated drums stem. We previously tried BeatNet, but
#   it imports `madmom` at module load and madmom's PyPI release is
#   stuck at 0.16.1 (2018) which doesn't build on Python ≥3.10. The
#   only way to get a working madmom is git+URL, which is outside
#   the trust boundary of an opt-in installer running on user
#   machines. librosa's accuracy is ~80-90 % on cleaned drums, which
#   is good enough for the workflow.
# - `numpy<2`: librosa 0.10 still pins numpy<2; demucs has no upper
#   bound but is fine on 1.x.

demucs==4.0.1
librosa>=0.10,<0.12
# `torch==2.2.2` (and matching torchaudio): demucs 4.0.1 was
# released against torch 2.x and `demucs.audio.save_audio` started
# requiring an external `torchcodec` package on torch ≥2.7. Pinning
# 2.2.2 — the last version with the in-tree audio codec — sidesteps
# that without giving up Apple Silicon MPS support, which 2.2 has.
torch==2.2.2
torchaudio==2.2.2
numpy>=1.24,<2
soundfile>=0.12
# `coverage>=7` — workaround for a numba 0.65.1 import-time bug
# (`numba.misc.coverage_support` does `import coverage` and then
# `class NumbaTracer(coverage.types.Tracer):` at module load).
# Without an actual coverage package on sys.path, Python's PEP 420
# namespace lookup hands back an empty `coverage` module, the
# class statement raises `AttributeError: module 'coverage' has
# no attribute 'types'`, and every `import librosa` fails.
coverage>=7
